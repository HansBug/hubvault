# 00. 目标、边界与约束

## 1. 项目目标

本项目要设计一个“本地版、嵌入式、API-only”的版本化仓库系统，重点服务于如下场景：

- 本地保存模型权重、数据集、训练产物、索引文件等超大文件
- 需要版本管理、历史回滚、分支与合并能力
- 需要很强的数据一致性保障
- 需要纯 Python 包即可启动，不依赖外部服务
- 需要跨平台支持 Linux、macOS、Windows
- 使用体验希望接近 `huggingface_hub` 的 `HfApi`

## 2. 核心约束

- 不需要 workspace
- 不需要命令行界面
- 不需要兼容 git 工作区和 index 语义
- 不依赖 SQLite、Redis、PostgreSQL 或其他外部数据库
- 不依赖守护进程、后台服务或单独安装的服务端
- 允许内部包含可选 C/Rust 加速模块，但正确性不能依赖原生扩展

## 3. 第一优先级

第一优先级不是“完全像 git”，而是：

- 已提交数据绝不损坏
- 崩溃恢复后仓库仍然处于合法状态
- 任意历史版本都可校验、可读取、可回滚
- 释放空间时不能误删仍被引用的数据

## 4. 非目标

以下内容不作为首版目标：

- 兼容 git CLI
- 兼容 git hooks
- 完整复刻 Hugging Face Hub 的账户、权限、PR、discussion、webhook、space 等平台能力
- 多写者高并发事务系统
- 网络文件系统上的强一致保证
- 通用二进制文件内容级自动 merge

## 5. 成功标准

系统至少需要满足以下成功标准：

- 单机本地即可初始化一个仓库
- 可通过 Python API 创建 repo、branch、tag、commit
- 可对文件和目录做增删改
- 可列出历史提交、分支、树结构和文件信息
- 可回滚到任意历史版本
- 可执行 merge，并清楚返回冲突
- 可执行 verify/fsck
- 可执行 GC/compact 来回收不可达历史空间

## 6. 术语约定

- `Repo`：一个逻辑仓库，对应一个本地根路径
- `Ref`：分支或标签，指向某个 commit
- `Commit`：一次不可变版本快照
- `Tree`：目录树对象
- `File`：文件版本对象
- `Chunk`：大文件切分后的数据块
- `Pack`：多个 chunk 的顺序存储文件
- `Txn`：一次写事务的暂存目录与提交流程
- `GC`：不可达对象清理与 pack 重写

## 7. 设计原则

- 不可变优先：所有核心对象一经提交即不可变
- 追加优先：主写路径尽量采用 append-only
- 显式回收：历史空间释放由单独 GC 完成
- 校验优先：对象、索引、pack 都要具备自校验能力
- 协议先行：先定义磁盘格式和事务协议，再谈实现细节
- 跨平台保守：只依赖三平台都稳定可用的文件系统语义
